    Awesome Integration ¶

    > 精选的系统集成软件、模式和资源列表.

    系统集成是将不同的 IT 系统（组件）链接在一起以在功能上作为一个整体进行协作的过程.

    Projects¶

    API Management¶

    API 管理解决方案. - Akana API Management Platform - 加速数字化转型. 管理完整的 API 生命周期. 跨多个云部署. 并确保合规. - Anypoint Platform - 在 Mulesoft 的单一平台中结合 API 管理和集成功能. - Amazon API Management - API 管理策略允许您以安全且可扩展的方式监控和管理 API. - Amplify API Management Platform - 来自 Axway 的 API 管理开放平台. - Apigee - 来自 Google 的 API 管理解决方案. - Azure API Management - 适用于所有环境中 API 的混合多云管理平台. - DigitMarket API Manager - 一个完整的软件包，可帮助管理您的 API 并将它们变成工具，由 Torry Harris 推动您的业务向前发展. - Gravitee.io API Management - 灵活、轻量级且速度极快的开源解决方案. - IBM API Connect - 一个完整、直观且可扩展的 API 平台，可让您跨云创建、公开、管理 API 并从中获利. - Kong Enterprise ——服务连接平台. - Layer7 API Management - 来自 Broadcom 的持续 API 管理. - Red Hat 3scale API Management - 云原生开发变得更简单、更灵活. - Sensedia API Management - 全生命周期API管理平台. - TIBCO Cloud Mashery - 可随处部署、随处管理 API 的云原生 API 平台. - Tyk API Management - 连接、构建和控制. - webMethods API - 用于全生命周期 API 管理的 Software AG 产品是 webMethods API 管理. 平台（用于本地部署）和 webMethods.io API（用于云）. - WSO2 API Manager - 完全开源的 API 管理平台.

    API Design¶

    API 设计、文档和生命周期自动化工具. - OpenAPI Diff - 比较两个 OpenAPI 规范 (3.x) 并将差异呈现为 HTML 纯文本或 Markdown 文件. - OpenAPI Generator - 根据 OpenAPI 规范（支持 2.0 和 3.0），允许自动生成 API 客户端库（SDK 生成）、服务器存根、文档和配置. - OpenAPI Style Validator - 可自定义的样式验证器，可确保您的 OpenAPI 规范符合您组织的标准. - Swagger Editor - 在第一个完全致力于基于 OpenAPI 的 API 的开源编辑器上设计、描述和记录您的 API. - Zally - OpenAPI 规范的 linter.

    API Gateway¶

    API 网关. - Ambassador Edge Stack - Kubernetes 原生 API 网关. - Apache APISIX - 动态、实时、高性能的 API 网关. - Gloo Edge - 基于 Envoy 代理的 API 网关. - Goku API Gateway - 基于Golang的微服务网关，实现高性能动态路由、服务编排、多租户管理、API访问控制等. - Kong API Gateway - 云原生、平台无关、可扩展的 API 网关以其高性能和通过插件的可扩展性而著称. - KrakenD API Gateway - 带有中间件的超高性能 API 网关. - Ocelot - .NET API 网关. - Spring Cloud Gateway - 构建在 Spring 生态系统之上的 API 网关，包括：Spring 5、Spring Boot 2 和 Project Reactor. - Tyk API Gateway - 开源企业 API 网关，支持 REST、GraphQL、TCP 和 gRPC 协议.

    API Testing¶

    API 测试工具. - API 客户端 - curl - 用于通过 URL 传输数据的命令行工具和库. - curlie - curl 的前端，增加了 httpie 的易用性，同时不影响功能和性能. - HTTPie - 命令行 HTTP 客户端，远比 curl 对开发人员友好. - Insomnia - 跨平台 HTTP 和 GraphQL 客户端. - Postman - 用于发出 HTTP 请求、生成模拟服务器和制作 API 文档的工具. - SoapUI - 用于 SOAP 和 REST 的开源 Web 服务测试应用程序. - MQ 客户 - JMSToolBox - 通用 JMS 客户端，能够与最多数量的基于 JMS 的消息代理进行交互. - kcat - Apache Kafka 的通用非 JVM 生产者和消费者，将其视为 Kafka 的 netcat. 以前称为kafkacat. - Offset Explorer - 用于管理和使用 Apache Kafka 集群的 GUI 应用程序. 以前称为 Kafka 工具. - 模拟工具 - Microcks - 用于 API 模拟和测试的开源 Kubernetes 本机工具. 支持 AsyncAPI、OpenAPI、Postman Collections 等. - Mockable - 一个简单的可配置服务，用于模拟 RESTful API 或 SOAP Web 服务. - Mockoon - 设计和运行模拟 REST API 的最简单快捷的方法. - MockServer - 轻松模拟您通过 HTTP 或 HTTPS 集成的任何系统. - Mocky - 一项免费且无限制的在线服务，允许生成自定义 HTTP 响应. - Prism - 一个开源 HTTP 模拟服务器，可以模仿您的 API 的行为，就好像您已经构建了它一样. - WireMock - 模拟您的 API 以进行快速、稳健和全面的测试. - 测试工具和框架 - Apache JMeter - 一个基于 Java 的开源负载测试工具. - Karate - 开源工具，可将 API 测试自动化、模拟、性能测试甚至 UI 自动化整合到一个统一的框架中. - REST Assured - 用于轻松测试 REST 服务的 Java DSL.

    BRE¶

    业务规则引擎和业务规则管理系统. - Drools - 开源工具，完全运行时支持一致性级别 3 的决策模型和符号 (DMN) 模型，以及用于核心开发的 Eclipse IDE 插件. - Easy Rules - 一个简单的 Java 规则引擎. - NxBRE - .NET 平台的开源规则引擎. - OpenL Tablets - 开源业务规则引擎（BRE、BRMS）和决策管理系统.

    BPM¶

    业务流程管理解决方案. - Appian BPM Suite - 平台将低代码开发与流程管理相结合 允许 IT 和公民开发人员构建以流程为中心和以案例为中心的应用程序. - IBM Business Automation Workflow - 一个全面的业务流程管理平台. - Oracle BPM Suite - 用于开发、管理和使用以业务流程为中心的业务应用程序的集成环境. - Pega Platform - 结合业务流程管理和机器人流程自动化 (RPA) 平台以及 Pegasystems 的高级劳动力分析. - SAP Process Orchestration - 业务流程管理产品和应用程序基础架构. - TIBCO BPM Enterprise - 具有流程自动化、流程文档、人力资本管理、流程模式和预测性运营分析功能的业务流程管理平台.

    Data Mapping Solution¶

    数据映射解决方案. - AltasMap - 具有基于 Web 的交互式用户界面的数据映射解决方案，可简化 Java、XML、CSV 和 JSON 数据源之间的集成配置. - JOLT - 用 Ja​​va 编写的 JSON 到 JSON 转换库，其中转换的“规范”本身就是一个 JSON 文档. - JSLT - 一个完整的 JSON 查询和转换语言. 语言设计的灵感来自 jq、XPath 和 XQuery.

    ESB¶

    企业服务总线解决方案. - Anypoint Platform - 在 Mulesoft 的单一平台中结合 API 管理和集成功能. - Apache ServiceMix - 一个灵活的开源集成容器，统一了 Apache ActiveMQ、Camel、CXF 和 Karaf 的特性和功能. - ArcESB - 跨应用程序同步数据，与合作伙伴无缝集成，或使数据可公开访问. - IBM App Connect - 将应用程序连接在一起，无论它们支持何种消息格式或协议. 以前称为 IBM 集成总线. - NServiceBus - The most developer-friendly service bus for .NET. - Oracle Service Bus - 通过连接、虚拟化和管理服务和应用程序之间的交互，将复杂而脆弱的架构转变为敏捷的集成网络. - Oracle SOA Suite - 使系统开发人员能够设置和管理服务，并将它们编排成复合应用程序和业务流程. - Red Hat Fuse - 分布式云原生集成平台. - Software AG webMethods Integration Server - 更快地集成任何东西. - TIBCO BusinessWorks - 实施混合集成的企业模式. - UltraESB - 唯一支持零复制代理的 ESB，利用直接内存访问 (DMA) 和发送文件系统调用以及非阻塞 IO 实现极端性能水平. - WSO2 Enterprise Integrator - 一个以 API 为中心、云原生的分布式集成平台.

    ETL¶

    ETL/ELT 和数据集成工具. - Apache NiFi - Apache NiFi 是一个集成的数据物流平台，用于自动化不同系统之间的数据移动. - CloverETL - 数据集成软件套件，用于数据迁移和数据仓储，以及将数据输入商业智能和报告应用程序. - IBM DataStage - 一个数据集成平台，可帮助企业理解、清理、转换和交付可信信息. - Informatica PowerCenter - 提供支持大数据和云分析的企业数据集成和管理软件. - Microsoft SSIS - SQL Server 集成服务是一个用于构建企业级数据集成和数据转换解决方案的平台. - Oracle Data Integrator - 一个全面的数据集成平台，涵盖所有数据集成需求，包括批量加载、集成流程和支持 SOA 的数据服务. - Pentaho Data Integration - 使用户能够使用可视化工具从任何来源摄取、混合、清理和准备不同的数据，以消除编码和复杂性. - SAS Data Management - 帮助转换、集成、治理和保护数据，同时提高其整体质量和可靠性. - Stitch - 一个简单、强大的 ETL 服务，适用于各种规模的企业，包括企业在内. Stitch 在可扩展、容错的云平台上运行，集成了来自数十个不同来源的数据. - Talend Data Integration - 将来自多个来源的数据汇集在一起​​，为您的所有业务决策提供支持.

    Integration Frameworks¶

    基于已知企业集成模式的集成框架. - Apache Camel - 通过企业集成模式将不同的传输 API 粘合在一起. - Ballerina - 一种开源编程语言，可以更轻松地使用、组合和创建网络服务. - Spring Integration - 扩展 Spring 编程模型以支持著名的企业集成模式.

    iPaaS¶

    Enterprise Integration Patterns¶

    来自 Gregor Hohpe 和 Bobby Woolf 一本书的模式. - Aggregator - 我们如何组合单独但相关的消息的结果，以便将它们作为一个整体进行处理？
    - Canonical Data Model - 在集成使用不同数据格式的应用程序时，如何最大限度地减少依赖性？ - Channel Adapter - 如何将应用程序连接到消息传递系统，以便它可以发送和接收消息？ - Channel Purger - 如何避免频道上的“剩余”消息干扰测试或运行系统？ - Claim Check - 如何在不牺牲信息内容的情况下，减少系统发送的消息数据量？
    - Command Message - 如何使用消息传递来调用另一个应用程序中的过程？
    - Competing Consumers - 消息客户端如何同时处理多条消息？ - Composed Message Processor - 在处理由多个元素组成的消息时，如何维护整体消息流，每个元素可能需要不同的处理？
    - Content Enricher - 如果消息发起者没有可用的所有必需数据项，我们如何与另一个系统通信？
    - Content Filter - 当您只对少数数据项感兴趣时，如何简化处理大消息的过程？
    - Content-Based Router - 我们如何处理单个逻辑功能（例如库存检查）的实现分布在多个物理系统中的情况？
    - Control Bus - 我们如何有效管理分布在多个平台和广泛地理区域的消息传递系统？
    - Correlation Identifier - 收到回复的请求者如何知道这是针对哪个请求的回复？ - Datatype Channel - 应用程序如何发送数据项，以便接收者知道如何处理它？ - Dead Letter Channel - 消息系统将如何处理它无法传递的消息？
    - Detour - 如何通过中间步骤路由消息以执行验证、测试或调试功能？ - Document Message - 如何使用消息传递在应用程序之间传输数据？ - Durable Subscriber - 订阅者如何避免在不收听消息时丢失消息？ - Dynamic Router - 如何在保持其效率的同时避免路由器对所有可能目的地的依赖？ - Envelope Wrapper - 现有系统如何参与对消息格式（例如消息头字段或加密）提出特定要求的消息交换？
    - Event Message - 如何使用消息传递将事件从一个应用程序传输到另一个应用程序？ - Event-Driven Consumer - 应用程序如何在消息可用时自动使用它们？ - Format Indicator - 如何设计消息的数据格式以允许未来可能发生的变化？ - Guaranteed Delivery - 即使消息传递系统出现故障，发件人如何确保消息会被传递？ - Idempotent Receiver - 消息接收者如何处理重复消息？ - Invalid Message Channel - 消息接收者如何优雅地处理接收毫无意义的消息？ - Message - 通过消息通道连接的两个应用程序如何交换一条信息？ - Message Dispatcher - 单个通道上的多个消费者如何协调他们的消息处理？ - Message Expiration - 发件人如何指示何时应将消息视为过时并因此不应处理？ - Message Translator - 使用不同数据格式的系统如何使用消息传递相互通信？
    - Message Broker - 如何将消息的目的地与发送方分离并保持对消息流的中央控制？ - Message Bus - 什么是使单独的应用程序能够协同工作，但以解耦的方式使应用程序可以轻松添加或删除而不影响其他应用程序的架构？
    - Message Channel - 一个应用程序如何使用消息传递与另一个应用程序通信？ - Message Endpoint - 应用程序如何连接到消息通道以发送和接收消息？ - Message Filter - 组件如何避免接收无趣的消息？ - Message History - 我们如何有效地分析和调试松散耦合系统中的消息流？
    - Message Router - 您如何解耦各个处理步骤，以便根据一组条件将消息传递到不同的过滤器？ - Message Sequence - 消息如何传输任意大量的数据？ - Message Store - 我们如何在不干扰消息系统松散耦合和瞬态特性的情况下报告消息信息？
    - Messaging Bridge - 如何连接多个消息系统，以便在一个系统上可用的消息也可在其他系统上使用？ - Messaging Gateway - 您如何封装从应用程序的其余部分对消息传递系统的访问？ - Messaging Mapper - 如何在域对象和消息传递基础设施之间移动数据，同时保持两者相互独立？ - Normalizer - 如何处理语义相同但以不同格式到达的消息？ - Pipes and Filters - 我们如何在保持独立性和灵活性的同时对消息进行复杂的处理？
    - Point-to-Point Channel - 呼叫者如何确定只有一位接收者会收到文件或执行呼叫？ - Polling Consumer - 当应用程序准备好时，应用程序如何使用消息？
    - Process Manager - 当所需的步骤在设计时可能未知并且可能不是连续的时，我们如何通过多个处理步骤路由消息？ - Publish-Subscribe Channel - 发送方如何向所有感兴趣的接收方广播一个事件？ - Recipient List - 我们如何将消息路由到动态指定的收件人列表？ - Request-Reply - 当应用程序发送消息时，它如何从接收者那里得到响应？ - Resequencer - 我们如何才能将相关但无序的消息流恢复到正确的顺序？ - Return Address - 回复者如何知道将回复发送到哪里？ - Routing Slip - 当步骤序列在设计时未知并且可能因每条消息而异时，我们如何通过一系列处理步骤连续路由消息？
    - Scatter-Gather - 当一条消息需要发送给多个收件人时，您如何维护整个消息流，每个收件人都可能发送回复？
    - Selective Consumer - 消息消费者如何选择它希望接收的消息？
    - Service Activator - 应用程序如何设计可通过各种消息传递技术和非消息传递技术调用的服务？ - Smart Proxy - 如何在向请求者指定的返回地址发布回复消息的服务上跟踪消息？ - Splitter - How can we process a message if it contains multiple elements, each of which may have to be processed in a different way?
    - Test Message - 但是，如果一个组件正在积极处理消息，但由于内部故障而导致外发消息出现乱码，会发生什么情况？
    - Transactional Client - 客户端如何通过消息传递系统控制其事务？
    - Wire Tap - 您如何检查在点对点通道上传输的消息？
